Content Studio is Enonic XP's content-management UI — the editor-facing application for creating, organising, and publishing content. This reference covers how the UI works for content managers and front-end developers. Tutorials and getting-started guides live on https://developer.enonic.com[developer.enonic.com] and build on top of this content.

NOTE: Content Studio can be installed from https://market.enonic.com/vendors/enonic/content-studio[Enonic Market].

== Intro videos

A collection of short videos demonstrating Content Studio's features and capabilities is available on the https://www.enonic.com/resources/content-studio-videos[Enonic site].

== Overview

Content lives in <<projects#,projects>> — top-level repositories with their own content tree, applications, and settings. <<layers#,Layers>> reuse, localise, or augment content across projects without duplicating trees.

The <<workbench#,Workbench>> is the in-tab application shell: a project bar on top, a menu bar on the left, and a main panel filling the rest. The default menuitem is the <<workbench/navigator#,Navigator>> — the content tree, search, and per-content actions.

Content is opened for editing in the <<editor#,Editor>>, which launches in a separate browser tab and provides a consistent interface for editing and previewing.

== Concepts

A few recurring concepts surface everywhere in the UI — in dialogs, widgets, toolbars, and URLs. For the low-level reference (schemas, field types, APIs), see the https://developer.enonic.com/docs/xp/stable/cms[Enonic XP CMS docs].

=== Content items and content types

A *content item* is a single piece of content — an article, image, page, or site. Every item has an identity (path, display name, id), a type, a set of field values, and a small envelope of metadata (security, schedule, workflow state).

A *content type* is the schema that defines an item's fields. Most are defined in application code; a handful ship with XP and have special behaviour:

* *Folder* — a grouping container with no data fields beyond display name and path.
* *Shortcut* — a path that redirects to another content item. Required *Target* field and optional name-value *Parameters*. Cannot use page templates.
* *Site* — the root of a website. Carries installed applications and gets a *Templates* folder as its first child. See <<sites#,Sites>>.
* *Page template* — a reusable page definition. See <<sites/page-templates#,Page templates>>.
* *Image* — backed by an image file, with cropping and focal-point tools. See <<editor/image#,Image>>.
* *Unstructured* — formless content used by applications to store free-form data; not editable in Content Studio beyond name, path, settings, and security.

=== Pages and rendering

Page rendering is driven by three collaborating concepts: the content item, the *page template* that describes how to render it, and the *page controller* (from an installed application) that supplies the HTML skeleton. Page building happens in the <<editor/page-editor#,Page editor>>.

=== Draft, master, and publishing

Content is always edited in the *draft* branch. Publishing copies the active draft version into the *master* branch, from which it becomes available to public consumers of the API.

*Workflow state* is the signal between editors that a draft is ready to publish — _In progress_, _Ready for publishing_, _Publish request_, _Published_, _Unpublished_. See <<workflow#,Workflow>> for the state machine and <<actions/publish#,Publish>> for the flow.

=== Versions

Every save, publish, unpublish, rename, or permissions change creates a new *version*. Versions can be compared and reverted from the Version history widget — see <<workbench/contextpanel/versions#,Versions>>.

=== Dependencies

Content items reference each other — pages embed images, shortcuts target content, text fields link to other items. *Dependencies* are tracked in both directions (inbound: who uses this; outbound: what does this use). Publishing, archiving, and unpublishing all check dependencies — see <<workbench/contextpanel/dependencies#,Dependencies>>.

=== Permissions

Each content item has its own security settings: a list of *principals* (users, groups, roles) and the *permission flags* they have on that item (_Read_, _Create_, _Modify_, _Delete_, _Publish_, _Read/Write permissions_). Permissions are inherited from parent content by default and can be overridden per-item. See <<workbench/permissions#,Permissions>>.

=== Archive

Deleting content is destructive. *Archiving* is the non-destructive alternative — items are moved to a separate archive storage where they can be browsed, restored, or permanently deleted later. See <<workbench/archive#,Archive>>.

=== Applications and widgets

An *application* is an installable unit of XP code that can contribute content types, page controllers, parts, layouts, translations, and UI widgets. Applications are attached to a site via its Applications field; some (like Juke and Content Studio+) contribute UI across all sites.

A *widget* is a small UI surface contributed by an application. The main host is the <<workbench/contextpanel#,Context panel>>; applications can also contribute <<workbench/preview#,preview modes>> and <<workbench#,menuitems>>.

=== Issues and publish requests

*Issues* are lightweight work items used to coordinate between editors — either a *task* (e.g. "spellcheck this article") or a *publish request* (review-and-publish handoff for users without publish permission, or for an approval flow). See <<issues#,Issues>> and <<issues/publish-request#,Publish request>>.

== Content Studio+

Content Studio itself is free and open source. An Enonic subscription unlocks *Content Studio+*, a companion application distributed on https://market.enonic.com/vendors/enonic/content-studio-plus[Enonic Market] that adds advanced editorial capabilities:

* <<ai#,AI Assistant (Juke)>> — in-context AI help for editors, including field-level assistance, variants, and project-level instructions.
* <<workbench/archive#,Archive browser>> — the dedicated *Archive* menuitem for browsing, restoring, and permanently deleting archived content.
* <<workbench/contextpanel/variants#,Content Variants>> — create and manage variant copies of a content item for A/B testing.
* <<layers#ai_translation,AI translation>> — automatic translation of inherited content when localising in a layer.
* <<layers#layers_panel,Layers widget>> — see the status of a content item across every layer in the system.

Without a valid license the Content Studio+ application can still be installed, but its UI contributions remain inactive. Pages that document a CS+ feature carry a *Content Studio+ feature* admonition at the top.

== Browser support

Content Studio works in all modern browsers. We are explicitly testing and supporting the latest versions of Google Chrome, Apple Safari, Mozilla Firefox, and Microsoft Edge.
