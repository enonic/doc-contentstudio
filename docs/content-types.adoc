= Built-in content types
:toc: right
:imagesdir: content-types/images

A few content types ship with Enonic XP itself and behave specially in Content Studio. Everything else — the custom content types that shape your editorial work — is defined by an application's schemas. For the schema reference (how custom content types are defined, field types, configuration flags, display-name and list-title expressions, etc.), see the https://developer.enonic.com/docs/xp/stable/cms/content-types[XP CMS content-types documentation].

image::new-content-dialog.png[New Content dialog, 723]

== Folder

A *Folder* is a plain grouping container — it has no data fields beyond the display name, path, and the standard Settings and Security sections. Folders are used to organise other content in the tree.

image::wizard-folder.png[Folder editor]

== Shortcut

*Shortcuts* create a content path that redirects to another content item, letting a deeply-nested content be reached via a short URL. For example, a shortcut can expose `/superhero/gotham` that redirects to `/superhero/posts/gotham-sure-is-a-big-town`.

Each shortcut has a required *Target* field (click the pencil icon to edit the target in place) and an optional list of name-value *Parameters* that are appended to the redirect URL.

image::wizard-shortcut.png[Shortcut editor]

NOTE: Shortcuts cannot use page templates — the redirect fires even if a matching template exists.

== Site

A *Site* is the root of a website and is usually created at the top level of the content tree. Sites have a description field and a selector for adding and configuring applications. A *Templates* folder is created automatically alongside every new site.

image::browse-template.png[Templates folder]

=== Applications

Applications provide the code behind a site. Some apps contribute components (parts, layouts, page controllers) used for page-building in the <<editor/page-editor#,Page editor>>; others just add functionality. A site needs at least one app that contributes a page controller before any content can be rendered as a page.

Apps are added to a site via the *Applications* selector in the site's edit view. Adding or removing an application requires the *Administrator* or *Content Manager Administrator* role.

image::wizard-applications.png[Adding applications to a site]

=== Application configuration

Each application added to a site appears with a pencil icon for editing its configuration (fields defined by the app) and an X icon for removing it. The available configuration options vary per application and only users with the *Administrator* or *Content Manager Administrator* role can modify them.

image::wizard-config.png[Application configuration dialog]

== Page template

A *Page template* is a reusable page definition that renders one or more content types as pages. Templates are created inside a site's automatically-provisioned *Templates* folder.

=== Creating a page template

Create a new *Page template* content inside the Templates folder. Two fields drive the template:

* *Supports* — the content types this template can render.
* *Controller* (the selector on the right of the editor) — the page component from an installed application that provides the HTML skeleton. Once a controller is picked, the Page editor becomes active and components can be placed into the controller's regions.

image::browse-templates-list.png[List of page templates]

=== Which template is used

When several templates support the same content type, the one that appears _first_ in the Templates folder is used by default. Individual content items can override the default via the *Page template* dropdown in the <<context-panel/components#_inspect_tab,Inspect tab>>.

image::wizard-template.png[Choosing a template for a content item]

=== Rendering content without a matching template

Content with no supporting template can still be rendered as a page — for example, rendering a *Folder* called _Articles_ as a listing of its children. Open the content in the editor, pick a page controller from the dropdown in the empty Page editor, and build the page from components as usual.

image::wizard-template-select.png[Selecting a controller for unsupported content]

== Templates folder

The *Templates* folder is a specialised folder that can only contain *Page template* content and is created automatically as a child of every site.

== Image

Uploading an image file creates an *Image* content item. Storing images as content means they are indexed for search, inherit the site's language and permissions, and can be referenced by ID from other content.

Image content items have fields for Caption, Artist, Copyright, Tags, and Text, and automatically pick up any Exif metadata from the source file. The underlying file can be swapped via the upload button or by dragging a new file onto the preview.

=== Cropping

The crop icon above the image dims the surrounding area and reveals the crop-editing tools. The zoom slider enlarges the preview; the image can be dragged to reposition. The aspect-ratio handle (circle with arrows at the bottom) changes the crop ratio. *Apply* returns the editor to normal mode.

image::image-crop.png[Cropping an image]

=== Focal point

Rendered images often need to be cropped to match a different aspect ratio than the original, which can cut off the subject. Setting a *focal point* tells the renderer which part of the image to always keep in frame.

Click the focal point icon, then click the part of the image that should stay centred, and *Apply*. The focal point is marked with a red circle in the editor.

== Unstructured

*Unstructured* content has no data form — it stores free-form data used by applications (for example, persisted form submissions). Unstructured content items can only be edited for their display name, path, settings, and security in Content Studio; the actual data is not visible in the UI unless an application provides a page controller that renders it.
