= Content Management
:toc: right
:imagesdir: images

Content Management covers the operations that apply _across_ content items once they exist — the editorial lifecycle rather than the shape of any single item.

These operations are available from both the <<navigator#,Content Navigator>> and the <<editor#,Content Editor>>, and they are gated by <<permissions#,permissions>>: whether a given button is shown, enabled, or disabled depends on the current user's rights for the selected content.

== In this chapter

<<actions#,Actions>>::
The toolbar and context-menu verbs editors use day-to-day — *New*, *Edit*, *Duplicate*, *Move*, *Sort*, *Preview*. Reference for what each action does, where to invoke it, and which dialogs it opens.

<<workflow#,Workflow>>::
The states a content item moves through between creation and publication — *In progress*, *Ready for publishing*, *Publish request*, *Published*, *Unpublished* — together with the actions that drive those transitions (*Publish*, *Unpublish*, *Mark as ready*, *Request publishing*).

<<permissions#,Permissions>>::
The per-content security model: principals (users, groups, roles), the granular permission flags (*Read*, *Create*, *Modify*, *Delete*, *Publish*, *Read/Write permissions*), inheritance, and the *Edit permissions* dialog.

<<variants#,Variants>>::
Create and manage variant copies of a content item for A/B testing. Requires <<content-studio-plus#,Content Studio+>>.

<<content-types#,Built-in content types>>::
The reference for the content types Enonic XP ships with — *Folder*, *Shortcut*, *Site*, *Page template*, *Image*, *Unstructured* — and how they behave in the Content Studio UI.

== Where this sits

Content Management assumes you already have content to manage. For the creation side, see <<navigator#,Navigator>> (finding and creating items) and <<editor#,Editor>> (editing a single item). For the schema and data structure behind a custom content type, see the https://developer.enonic.com/docs/xp/stable/cms[Enonic XP CMS docs].

Two closely-related chapters have their own top-level entries because they also have dedicated main-menu destinations in the Content Studio UI: <<archive#,Archive>> (browsing, restoring, and permanently deleting archived content) and <<issues#,Issues>> (tasks and publish requests).
